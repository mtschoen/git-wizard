# Gitea CI for git-wizard Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add Gitea Actions CI to `schoen/git-wizard` that builds cross-platform projects on Linux, builds the full solution and runs tests on Windows, and produces a multi-platform release artifact set on `v*` tag pushes.

**Architecture:** Two workflow YAML files under `.gitea/workflows/`. `ci.yml` runs `build-linux` (cross-platform csprojs only) and `test-windows` (full solution + `dotnet test`) in parallel on push-to-main and PRs. `release.yml` runs three jobs on tag push: `publish-cross` (CLI + Avalonia for win-x64/linux-x64/osx-x64), `publish-maui` (the existing `dotnet publish` zip flow), and `release` (creates Gitea release, uploads assets). A dedicated `ci-bot` Gitea user holds the PAT used by `release`.

**Tech Stack:** Gitea Actions (GitHub-Actions-compatible YAML), `actions/checkout@v4`, `actions/setup-dotnet@v4`, `actions/cache@v4`, `actions/upload-artifact@v4`, `actions/download-artifact@v4`, .NET 10 SDK, MAUI workload (Windows runner only), Gitea REST API for release/asset upload.

**Conventions for this plan:**
- The "tests" for CI workflows are **observed runs**: we push to a feature branch (or trigger via `workflow_dispatch` for the release workflow) and watch the run pass. Each task has an explicit verification step pointing at the run UI URL.
- We commit and push frequently. Each task ends with a commit + push so the next task can observe a real run.
- The plan produces ONE PR (`ci/setup` branch) that lands all the CI files in a logical sequence. We don't need branch protection until the workflow is proven, so it's the last (manual) phase.
- Anything the user must run by hand (provisioning the bot user, adding secrets, configuring branch protection) is called out as a **MANUAL** step. Do not attempt to do these steps from the agent.
- The spec is at `docs/superpowers/specs/2026-04-28-gitea-ci-design.md`. Refer to it for "why" decisions; this plan is "how".

## Phase 1: `ci.yml` with `build-linux` job

### Task 1.1: Create `ci/setup` branch and the workflow file with the Linux job

**Files:**
- Create: `.gitea/workflows/ci.yml`

- [ ] **Step 1: Create branch and switch**

  ```bash
  git checkout main
  git pull --ff-only
  git checkout -b ci/setup
  ```

  Expected: switched to a new branch `ci/setup`.

- [ ] **Step 2: Create `.gitea/workflows/ci.yml` with the Linux job only**

  ```yaml
  name: CI

  on:
    push:
      branches: [main]
    pull_request:
      branches: [main]

  concurrency:
    group: ci-${{ github.ref }}
    cancel-in-progress: ${{ github.event_name == 'pull_request' }}

  jobs:
    build-linux:
      name: Build (Linux, cross-platform projects)
      runs-on: ubuntu-latest
      timeout-minutes: 30

      steps:
        - name: Checkout
          uses: actions/checkout@v4

        - name: Set up .NET 10 SDK
          uses: actions/setup-dotnet@v4
          with:
            dotnet-version: 10.0.x

        - name: Cache NuGet packages
          uses: actions/cache@v4
          with:
            path: ~/.nuget/packages
            key: linux-nuget-${{ hashFiles('**/*.csproj') }}
            restore-keys: |
              linux-nuget-

        - name: Restore (cross-platform projects)
          run: |
            for proj in \
              GitWizard/GitWizard.csproj \
              git-wizard/git-wizard.csproj \
              GitWizardUI.ViewModels/GitWizardUI.ViewModels.csproj \
              GitWizardAvalonia/GitWizardAvalonia.csproj \
              GitWizardTests/GitWizardTests.csproj
            do
              dotnet restore "$proj"
            done

        - name: Build (cross-platform projects)
          run: |
            for proj in \
              GitWizard/GitWizard.csproj \
              git-wizard/git-wizard.csproj \
              GitWizardUI.ViewModels/GitWizardUI.ViewModels.csproj \
              GitWizardAvalonia/GitWizardAvalonia.csproj \
              GitWizardTests/GitWizardTests.csproj
            do
              dotnet build "$proj" -c Release --no-restore
            done
  ```

- [ ] **Step 3: Commit and push**

  ```bash
  git add .gitea/workflows/ci.yml
  git commit -m "ci: add Linux build job for cross-platform projects

Builds GitWizard core, CLI, ViewModels, Avalonia, and the test project
on the ubuntu-latest runner. MAUI is intentionally skipped on Linux
because GitWizardUI.csproj has no TargetFrameworks outside Windows/macOS.

See docs/superpowers/specs/2026-04-28-gitea-ci-design.md.

Co-Authored-By: Claude Opus 4.7 (1M context) <noreply@anthropic.com>"
  ```

  Expected: branch published; CI run kicks off automatically because the workflow file is now on the branch.

## Phase 3: `release.yml` — start with `publish-cross`

The release workflow is harder to test on a feature branch because it triggers on tag push. Strategy: include `workflow_dispatch` from the start so the workflow can be run on-demand against the `ci/setup` branch for end-to-end verification, and remove `workflow_dispatch` (or leave it for ops convenience) once the PR merges.

### Task 3.1: Create `release.yml` with `publish-cross` only

**Files:**
- Create: `.gitea/workflows/release.yml`

- [ ] **Step 1: Create the file**

  ```yaml
  name: Release

  on:
    push:
      tags: ['v*']
    workflow_dispatch:
      inputs:
        version:
          description: 'Version (without leading v) — used for filenames when run manually'
          required: true
          default: '0.0.0-test'

  jobs:
    publish-cross:
      name: Publish CLI + Avalonia (cross-platform)
      runs-on: ubuntu-latest
      timeout-minutes: 30

      strategy:
        fail-fast: false
        matrix:
          rid: [win-x64, linux-x64, osx-x64]

      steps:
        - name: Checkout
          uses: actions/checkout@v4

        - name: Set up .NET 10 SDK
          uses: actions/setup-dotnet@v4
          with:
            dotnet-version: 10.0.x

        - name: Cache NuGet packages
          uses: actions/cache@v4
          with:
            path: ~/.nuget/packages
            key: linux-nuget-${{ hashFiles('**/*.csproj') }}
            restore-keys: |
              linux-nuget-

        - name: Resolve version
          id: ver
          run: |
            if [ "${{ github.event_name }}" = "push" ]; then
              VERSION="${GITHUB_REF_NAME#v}"
            else
              VERSION="${{ inputs.version }}"
            fi
            echo "version=$VERSION" >> "$GITHUB_OUTPUT"
            echo "Resolved version: $VERSION"

        - name: Publish CLI
          run: |
            dotnet publish git-wizard/git-wizard.csproj \
              -c Release \
              -r ${{ matrix.rid }} \
              --self-contained \
              -p:PublishSingleFile=true \
              -p:IncludeNativeLibrariesForSelfExtract=true \
              -o publish/cli/${{ matrix.rid }}

        - name: Publish Avalonia
          run: |
            dotnet publish GitWizardAvalonia/GitWizardAvalonia.csproj \
              -c Release \
              -r ${{ matrix.rid }} \
              --self-contained \
              -p:PublishSingleFile=true \
              -p:IncludeNativeLibrariesForSelfExtract=true \
              -o publish/avalonia/${{ matrix.rid }}

        - name: Zip artifacts
          run: |
            VERSION="${{ steps.ver.outputs.version }}"
            mkdir -p artifacts
            (cd publish/cli/${{ matrix.rid }} && zip -r "../../../artifacts/git-wizard-${VERSION}-${{ matrix.rid }}.zip" .)
            (cd publish/avalonia/${{ matrix.rid }} && zip -r "../../../artifacts/GitWizardAvalonia-${VERSION}-${{ matrix.rid }}.zip" .)
            ls -la artifacts/

        - name: Upload artifacts
          uses: actions/upload-artifact@v4
          with:
            name: cross-${{ matrix.rid }}
            path: artifacts/*.zip
            if-no-files-found: error
  ```

- [ ] **Step 2: Commit and push**

  ```bash
  git add .gitea/workflows/release.yml
  git commit -m "ci: add release workflow with cross-platform publish job

Publishes CLI and Avalonia for win-x64/linux-x64/osx-x64 on tag push
(v*) or manual workflow_dispatch. Each RID produces a self-contained
single-file zip uploaded as a workflow artifact. The release-creation
job is added in a follow-up commit.

The workflow_dispatch input lets us verify the workflow on the
ci/setup branch without tagging.

Co-Authored-By: Claude Opus 4.7 (1M context) <noreply@anthropic.com>"
  git push
  ```

  ```yaml
    publish-maui:
      name: Publish MAUI (Windows)
      runs-on: windows-latest
      timeout-minutes: 45

      steps:
        - name: Checkout
          uses: actions/checkout@v4

        - name: Set up .NET 10 SDK
          uses: actions/setup-dotnet@v4
          with:
            dotnet-version: 10.0.x

        - name: Cache NuGet packages
          uses: actions/cache@v4
          with:
            path: ~\.nuget\packages
            key: windows-nuget-${{ hashFiles('**/*.csproj') }}
            restore-keys: |
              windows-nuget-

        - name: Cache MAUI workload manifests
          uses: actions/cache@v4
          with:
            path: |
              ~\.dotnet\sdk-manifests
              ~\.dotnet\workloads
            key: windows-maui-${{ hashFiles('**/*.csproj') }}
            restore-keys: |
              windows-maui-

        - name: Install MAUI workload
          shell: pwsh
          run: dotnet workload install maui-windows

        - name: Resolve version
          id: ver
          shell: pwsh
          run: |
            if ($env:GITHUB_EVENT_NAME -eq 'push') {
              $version = $env:GITHUB_REF_NAME -replace '^v', ''
            } else {
              $version = '${{ inputs.version }}'
            }
            "version=$version" >> $env:GITHUB_OUTPUT
            Write-Host "Resolved version: $version"

        - name: Validate tag matches ApplicationDisplayVersion (push only)
          if: github.event_name == 'push'
          shell: pwsh
          run: |
            [xml]$csproj = Get-Content GitWizardUI/GitWizardUI.csproj
            $appVer = ($csproj.Project.PropertyGroup.ApplicationDisplayVersion | Where-Object { $_ }) -as [string]
            $tagVer = '${{ steps.ver.outputs.version }}'
            if ($appVer -ne $tagVer) {
              Write-Error "Tag version '$tagVer' does not match GitWizardUI.csproj ApplicationDisplayVersion '$appVer'. Bump the csproj or move the tag."
              exit 1
            }
            Write-Host "Version match: $tagVer"

        - name: Publish MAUI
          shell: pwsh
          run: |
            dotnet publish GitWizardUI/GitWizardUI.csproj `
                -f net10.0-windows10.0.19041.0 -c Release

        - name: Upload MAUI zip
          uses: actions/upload-artifact@v4
          with:
            name: maui-windows
            path: Releases/GitWizardUI-*.zip
            if-no-files-found: error
  ```

## Phase 5: Add `release` job

### Task 5.1: Append `release` job to `release.yml`

**Files:**
- Modify: `.gitea/workflows/release.yml`

**Note about MAUI zip filename:** The existing MSBuild target uses `$(ApplicationDisplayVersion)` from the csproj for the zip name, *not* the workflow's resolved version. On a real `v*` tag push the version-match guard ensures these are equal. On `workflow_dispatch` they may differ — the release job below uploads whatever the publish jobs produced, by glob, which is correct.

- [ ] **Step 1: Add the job**

  ```yaml
    release:
      name: Create Gitea release
      runs-on: ubuntu-latest
      needs: [publish-cross, publish-maui]
      if: github.event_name == 'push'
      timeout-minutes: 10

      steps:
        - name: Checkout
          uses: actions/checkout@v4

        - name: Resolve version
          id: ver
          run: |
            VERSION="${GITHUB_REF_NAME#v}"
            echo "version=$VERSION" >> "$GITHUB_OUTPUT"
            if [[ "$GITHUB_REF_NAME" == *-* ]]; then
              echo "prerelease=true" >> "$GITHUB_OUTPUT"
            else
              echo "prerelease=false" >> "$GITHUB_OUTPUT"
            fi

        - name: Download all artifacts
          uses: actions/download-artifact@v4
          with:
            path: downloaded

        - name: Stage assets
          run: |
            mkdir -p assets
            find downloaded -name '*.zip' -exec cp -v {} assets/ \;
            ls -la assets/

        - name: Create release
          id: create
          env:
            GITEA_TOKEN: ${{ secrets.GITEA_TOKEN }}
          run: |
            BODY=$(jq -nc \
              --arg tag "$GITHUB_REF_NAME" \
              --arg name "$GITHUB_REF_NAME" \
              --argjson pre ${{ steps.ver.outputs.prerelease }} \
              '{tag_name:$tag, name:$name, draft:false, prerelease:$pre}')
            RESPONSE=$(curl -fsSL -X POST \
              -H "Authorization: token $GITEA_TOKEN" \
              -H "Content-Type: application/json" \
              -d "$BODY" \
              "https://gitea.llamabox.internal/api/v1/repos/${GITHUB_REPOSITORY}/releases")
            RELEASE_ID=$(echo "$RESPONSE" | jq -r '.id')
            echo "Release id: $RELEASE_ID"
            echo "release_id=$RELEASE_ID" >> "$GITHUB_OUTPUT"

        - name: Upload assets
          env:
            GITEA_TOKEN: ${{ secrets.GITEA_TOKEN }}
            RELEASE_ID: ${{ steps.create.outputs.release_id }}
          run: |
            for asset in assets/*.zip; do
              echo "Uploading $asset"
              curl -fsSL -X POST \
                -H "Authorization: token $GITEA_TOKEN" \
                -H "Content-Type: application/octet-stream" \
                --data-binary "@$asset" \
                "https://gitea.llamabox.internal/api/v1/repos/${GITHUB_REPOSITORY}/releases/${RELEASE_ID}/assets?name=$(basename "$asset")"
            done
  ```

  Notes on the YAML above:
  - `if: github.event_name == 'push'` — the release job only runs on real tag pushes, not on `workflow_dispatch`. This is intentional: dispatching against `ci/setup` should produce artifacts for inspection but not create real releases.
  - Asset upload uses `Content-Type: application/octet-stream` with `--data-binary @file` and `?name=...` query param, which is Gitea's documented release-asset API.
  - `${GITHUB_REPOSITORY}` is `schoen/git-wizard` automatically.
